# ECHO 工作日志规则

`docs/worklogs` 保存每次“有实质修改的工作会话”的结构化摘要，帮助新任务快速接手和用户
复盘方法。它不保存完整聊天记录，也不替代 Git diff、阶段验收文档或原始测试文件。

## 1. 什么时候新建

以下任一发生时，新建一份日志：

- 修改了源码、配置、构建脚本或长期文档。
- 完成一次具有结论的硬件测试或故障定位。
- 合并阶段分支、创建提交/标签或恢复用户改动。
- 做出会影响后续架构、接线或验收的决定。

只有只读问答、没有新结论的重复检查，不必新建日志。

文件名：

```text
YYYY-MM-DD_<short-topic>.md
```

同一天同主题有多个独立会话时，在 topic 后加 `-2`、`-3`，不要覆盖前一份。

## 2. 必填内容

```markdown
# <日期> <主题>

性质：normal / retrospective

## 目标与范围
- 本次要完成什么
- 明确不做什么

## 开始状态
- 正式工程路径
- 分支、commit、tag
- dirty/staged 文件
- 相关备份或 stash

## 修改
- 文件清单
- 关键设计决定及原因
- 接口或接线变化

## 硬件状态
- 板卡、供电、接线、外设版本
- 测试时哪些设备未连接

## 验证
- 执行的命令/方法
- 时长和条件
- 关键指标和结果
- 哪些检查没有执行

## 问题与判断
- 现象
- A/B 证据
- 能证明什么、不能证明什么

## 风险与下一步
- 未解决风险
- 下一步及进入条件

## 结束状态
- 最终 commit/tag
- push 状态
- 保留的用户改动、stash 和备份
```

## 3. 证据规则

- 写“编译通过”必须有对应构建；写“板上通过”必须真的烧录并运行。
- 实时测试记录固件、启动方式、是否保持调试器连接、时长和是否中途暂停。
- 表格中的帧数、错误数和栈余量应来自保存的日志或诊断快照。
- 没有原始数据时写“未留存”，不得根据印象补数字。
- 事后根据 Git 和阶段文档补写的历史日志标记 `retrospective`。
- 旧日志不随新代码变化而重写；若发现错误，增加带日期的勘误说明。

## 4. 什么不进 Git

- 完整聊天记录。
- 大型串口原始流、重复构建日志、临时截图。
- 含本机隐私、令牌或账号的数据。
- 可由构建重新生成的 AXF、HEX、对象文件和网页缓存。

必要的大文件保存在仓库外，在日志中写路径、生成时间、固件提交和 SHA-256。路径若只是临时
位置，必须标注可能失效。

## 5. 与其他文档的关系

- `PROJECT_STATUS.md`：只保留当前权威状态，不写完整过程。
- 当前 Phase 文档：阶段范围、设计和验收标准/结果。
- `worklogs`：一次工作是怎样完成的、当时有什么证据和风险。
- `learning/DEBUGGING_PLAYBOOK.md`：按小模块保存可复用的现象、根因、排查、恢复和预防方法。
- `learning/INTEGRATION_PLAYBOOK.md`：登记引脚迁移、共享资源、初始化、降级和组合回归。
- `learning/MODULE_DEBUG_RECORD_TEMPLATE.md`：每个小模块完成前的强制记录模板。

完成工作时先写 worklog，再把仍然有效的结论提炼到 `PROJECT_STATUS.md`；不要把整份日志复制
进状态文件。

如果本次完成了一个可独立验收的小模块，结束前还必须：

1. 在调试手册索引中增加或更新唯一 module ID。
2. 按模板记录硬件身份、接线、owner、复现、诊断、A/B、修复和恢复方法。
3. 在集成手册登记物理映射、Timer/DMA/IRQ/任务/内存/ID、pin 迁移和共享冲突。
4. 分别记录构建、烧录、板测、故障测试、连续运行、两两组合和基线回归的真实状态。
5. 将 `not run`、`deferred` 和遗留风险保留在条目中。

未更新调试手册和集成手册时，worklog 完成不等于小模块完成或可集成完成。
